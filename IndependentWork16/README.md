# Самостійна робота №16

## Тема
Схема розподілу відповідальностей модуля

## Мета
Навчитися застосовувати принцип єдиної відповідальності (SRP) для декомпозиції складного модуля на менші, більш сфокусовані класи, а також візуалізувати розподіл відповідальностей за допомогою діаграми класів (UML).



## Контрольні питання

**1. Поясніть принцип єдиної відповідальності (SRP).**  
Клас повинен виконувати лише одну відповідальність, тобто мати одну причину для зміни. Це підвищує зрозумілість, тестованість та підтримуваність коду.

**2. Що таке “God Object” і як він порушує SRP?**  
“God Object” — це клас, який виконує занадто багато функцій (зберігає дані, робить бізнес-логіку, працює з БД, надсилає повідомлення тощо). Він порушує SRP, бо має багато причин для зміни, що ускладнює підтримку.

**3. Як декомпозиція допомагає дотримуватися SRP? Наведіть приклад.**  
Декомпозиція розбиває великий клас на менші, кожен з яких відповідає за одну задачу.  
Наприклад: `OrderProcessor` → розділений на `OrderValidator`, `OrderRepository`, `NotificationService`. Кожен клас відповідає за одну функцію.

**4. Як діаграми класів (UML) допомагають візуалізувати та аналізувати розподіл відповідальностей?**  
UML показує класи, їхні методи, атрибути та взаємозв’язки. Це дозволяє зрозуміти, хто за що відповідає, і перевірити дотримання SRP.



## Висновок
Застосування принципу єдиної відповідальності (SRP) дозволяє:  
- підвищити зрозумілість коду;  
- полегшити тестування окремих компонентів;  
- спростити підтримку та розширення програми.

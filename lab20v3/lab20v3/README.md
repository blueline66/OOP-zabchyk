# Звіт до лабораторної роботи Lab20

Метою цієї роботи було навчитися застосовувати принцип Single Responsibility Principle (SRP) у програмуванні. На початку ми зробили клас OrderProcessor, який порушував SRP, бо одночасно робив занадто багато речей: перевіряв замовлення, зберігав його, відправляв email і змінював статус. Такий підхід ускладнює підтримку та тестування коду.

Потім ми зробили рефакторинг. Створили окремі інтерфейси: IOrderValidator для перевірки замовлення, IOrderRepository для збереження та IEmailService для відправки повідомлень. Для кожного інтерфейсу зробили прості реалізації: OrderValidator, InMemoryOrderRepository та ConsoleEmailService.

Клас OrderService координує роботу цих компонентів через Dependency Injection. Він викликає перевірку замовлення, зберігає його та надсилає підтвердження клієнту. Таким чином, кожен клас відповідає лише за одну річ, і код став зрозумілим та легким для змін.

У методі Main ми перевірили роботу програми на валідному та невалідному замовленні. Це показало, що система працює правильно і окремі компоненти взаємодіють коректно.

# Висновок

Під час роботи ми зрозуміли, що розділення відповідальностей робить код простішим та зрозумілішим. Завдяки SRP та Dependency Injection, програма стала більш гнучкою, її легко підтримувати і тестувати.

# Звіт до лабораторної роботи №22: Принцип підстановки Лісков (LSP)
# 1. Аналіз помилки (Порушення LSP)
# Суть проблеми:
У початковому коді ми створили клас SavingsAccount (ощадний), який успадковувався від класу Account (звичайний).
Базовий клас (Account) мав чітке правило: якщо на балансі є гроші, їх можна зняти.
Похідний клас (SavingsAccount) змінив це правило: він заборонив зняття, якщо залишок на рахунку стане меншим за 1000, навіть якщо гроші там є.
# Чому це неправильно:
Це порушує принцип LSP через посилення передумов. Похідний клас став більш вибагливим, ніж базовий. Коли програма намагалася працювати з SavingsAccount так само як зі звичайним Account, вона отримувала помилку, якої не очікувала. Це означає, що ми не могли безболісно замінити батьківський клас на дочірній.

# 2. Виправлення (Рефакторинг)
# Рішення:
Ми змінили структуру класів. Замість того, щоб ощадний рахунок був "спадкоємцем" звичайного, ми створили для них спільного предка — абстрактний клас BankAccount.
# Як це працює:
Абстрактний клас BankAccount повідомляє системі, що операція зняття коштів (Withdraw) існує, але правила для неї кожен рахунок встановлює свої.
Тепер і звичайний, і ощадний рахунки знаходяться на одному рівні ієрархії.
Клієнтський код більше не думає, що "всі рахунки працюють однаково". Відмова у транзакції тепер сприймається програмою як правильна робота логіки, а не як збій.

# 3. Висновки
# Лабораторна робота показала важливість принципу LSP:

Наслідування — це відповідальність. Якщо ми створюємо спадкоємця, він повинен виконувати всі обіцянки батьківського класу. Не можна ламати поведінку, яку очікує програма від базового типу.
Спільна абстракція рятує. Якщо два класи схожі, але мають різні правила обмежень (як у нашому випадку), краще не успадковувати їх один від одного, а створити для них спільний абстрактний клас.
Результат. Дотримання LSP робить код передбачуваним і надійним, дозволяючи уникнути раптових помилок при розширенні програми.

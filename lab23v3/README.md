# Лабораторна робота №23
# Тема: ISP & DIP: рефакторинг і DI через конструктор.
# Мета: Застосувати принципи розділення інтерфейсу (ISP) та інверсії залежностей (DIP) для
рефакторингу існуючого коду, а також реалізувати Dependency Injection (DI) через конструктор
для зменшення зв’язаності та покращення тестування.

# Висновок
У початковій реалізації клас PaymentHandler напряму створював усі платіжні сервіси та залежав від конкретних класів, що порушувало принцип інверсії залежностей (DIP) і змушувало клієнта знати про всі методи оплати одночасно, навіть якщо використовувався лише один, що порушує принцип розділення інтерфейсів (ISP).

Після рефакторингу ми виділили вузькоспеціалізовані інтерфейси для кожного способу оплати та впровадили залежності через конструктор (Dependency Injection). Тепер PaymentHandler залежить лише від абстракцій, а не від конкретних класів, що підвищує гнучкість, спрощує тестування та дозволяє легко додавати нові платіжні методи без зміни існуючого коду.

Таким чином, дотримання принципів ISP та DIP зробило код чистим, масштабованим і підтримуваним.
